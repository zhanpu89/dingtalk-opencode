## 四、问题详情

### 维度 1：编码规范 ✅

- ✅ ESM 模块系统统一（`import` / `export`），无 `require` 混合
- ✅ 日志使用 `Logger` 框架，无裸 `console.log`
- ✅ 命名规范（camelCase 函数、PascalCase 类、UPPER_CASE 常量）
- ✅ 注释说明"为什么"而非"是什么"

### 维度 2：业务逻辑 ✅

| 检查项 | 结论 |
|--------|------|
| 强制重启流程完整 | ✅ `stopAllProjects → flush → spawn 脚本 → exit(0)` |
| 重启通知机制 | ✅ 标记文件桥接两个进程实例 |
| aiContext 无项目时重置 | ✅ 防止前一次项目切换污染 |
| 项目服务器 stdin 关闭 | ✅ 防止交互式弹窗挂起 |
| 看门狗 criticalAbort | ✅ 仅状态确认变更后触发，瞬态波动不中断 |
| checkSession 超时保护 | ✅ 15s 超时，防止阻塞看门狗 |

**CR-001 (P1)**: `defaultOpencode["baseUrl"]` 利用 TypeScript 编译期 `private` 仅作类型检查的漏洞，运行时仍可访问。建议改为 `public readonly` 或增加 getter：
```typescript
// 方案 A: 改为 public readonly
public readonly baseUrl: string;

// 方案 B: 增加 getter
getBaseUrl(): string { return this.baseUrl; }
```

### 维度 3：安全 ✅

- ✅ 无 SQL/命令注入风险（本模块无数据库）
- ✅ 子进程 stdin 已关闭，防止交互式输入泄露
- ✅ 敏感操作有 try-catch 保护（暴力重启/项目切换）
- ✅ `detached: true` 正确隔离子进程组

### 维度 4：性能 ✅

- ✅ 看门狗间隔 60s，健康检查 5s 超时，合理
- ✅ `Promise.race` 超时模式正确，不泄漏定时器
- ✅ `child.unref()` 防止父进程等待子进程

**CR-002 (P2)**: `/proc` 遍历回退逻辑在进程数多的系统上可能有性能开销。但仅在 `pkill` 不可用时（如 Alpine 镜像）触发，且重启脚本本身运行频率极低（仅暴力重启时），实际影响可忽略。

### 维度 5：可维护性 ✅

- ✅ 函数体均在合理长度内
- ✅ 关键算法有注释说明设计意图
- ✅ 错误处理统一（日志 + 用户通知）

**CR-003 (P2)**: `CI: "true"` 是常见做法但可能影响 npm 生命周期脚本行为。已确认项目 serve 场景下无副作用。
