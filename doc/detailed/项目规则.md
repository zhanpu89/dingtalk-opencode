## 第零部分：语言契约（Language Contract）

> **这是所有下游 Skill 必须首先读取的节。** coding-executor、code-reviewer
> 在执行任何操作前，必须先读取本节，确保所有输出与此契约一致。

| 契约编号 | 契约项 | 值 | 说明 |
|----------|--------|-----|------|
| LC-001 | 目标语言 | Node.js (TypeScript) | 所有代码必须使用 TypeScript 编写，ESM 模块 |
| LC-002 | 语言版本 | Node.js 20+ | 使用 Node.js 20+ 内置 API（fetch、AbortController），禁止 polyfill |
| LC-003 | Web 框架 | 无 | 本项目不使用 Web 框架，HTTP 调用使用原生 fetch |
| LC-004 | 数据访问层 | 无 | 无 ORM，持久化使用 JSON 文件 + fs 模块 |
| LC-005 | 包管理工具 | npm | 依赖声明和构建均使用 npm |
| LC-006 | 测试框架 | Vitest | 单元测试和集成测试均使用 Vitest |
| LC-007 | 日志框架 | 自定义 logger | 使用 `src/logger.ts`，禁止 console.log |
| LC-008 | 工具库偏好 | 标准库优先 | 优先使用 Node.js 内置 API，不引入 lodash 等通用工具库 |
| LC-009 | 构建产物 | tsc 编译 dist/ | 开发使用 tsx，生产使用 tsc 编译后 node 运行 |
| LC-010 | 代码分层模型 | 函数式模块 | 按职责拆分为独立文件，模块间函数引用，无 DI 容器 |

### 前端语言契约（不适用）

本项目为纯后端，无前端/小程序端。

## 第一部分：业务规则层（Business Rules）

### BR-GLOBAL：全局业务约束

| 规则编号 | 规则描述 | 适用范围 | 违反后果 |
|----------|----------|----------|----------|
| BR-G-001 | 消息串行：同一用户（sessionKey）的多条消息依次处理，前一条完成才执行后一条 | 消息排队 | 消息顺序错乱，AI 上下文混淆 |
| BR-G-002 | 重试限制：AI 处理异常最多重试 3 次，超限后发送失败提示不再继续 | AI 处理循环 | 无限重试浪费资源 |
| BR-G-003 | 发送失败不传播：钉钉消息发送异常内部吞掉，不中断主流程 | 消息发送 | 连锁崩溃导致用户收不到结果 |
| BR-G-004 | 项目隔离：不同项目之间的 AI 会话完全隔离，sessionKey 含 projectId 维度 | 会话管理 | 项目 A 上下文混入项目 B |

（来源：`消息调度与AI处理.md` §2 MSG-REG-01~MSG-REG-09）

### BR-AUTH：认证授权规则

| 规则编号 | 规则描述 | 实现方式 |
|----------|----------|----------|
| BR-A-001 | 只能访问 `projects.json` 中声明的项目，不允许自由文本指定任意路径 | config.ts 加载 + project-registry.ts 白名单校验 |
| BR-A-002 | 所有动态启动的项目路径必须经 realpath 解析后位于 ALLOWED_PROJECT_ROOTS 内 | server-manager.ts 启动校验 |

（来源：`服务管理与项目配置.md` §2 SVR-REG-01~SVR-REG-03）

### BR-DATA：数据完整性规则

| 规则编号 | 规则描述 | 涉及文件 |
|----------|----------|----------|
| BR-D-001 | 会话映射文件损坏时以空 Map 启动，不阻塞主进程 | `data/sessions.json` |
| BR-D-002 | 会话写入有 2s 防抖，避免高频 I/O | `data/sessions.json` |
| BR-D-003 | sessionKey 格式：`projectId:conversationId:senderId` | 内存 Map |

（来源：`会话与上下文持久化.md` §2 STO-REG-01~STO-REG-03）

## 第二部分：工程规则层（Engineering Rules）

### ER-FORBIDDEN：项目专属禁止项

| 禁止项 | 涉及模块 | 原因 | 正确方式 | 来源 |

| 禁止在 index.ts 中直接调用 opencode API | index.ts | 违反路由/编排分离原则 | 通过 ai-handler.ts 调用 | 架构决策 |
| 禁止在 ai-handler.ts 中直接操作文件 | ai-handler.ts | 数据访问应统一通过 session-store | 使用 sessionStore.get/set | `消息调度与AI处理.md` §13 |

| 禁止在 dingtalk.ts 中抛出异常 | dingtalk.ts | 发送失败不应影响主流程 | 内部 catch + logger.warn | `消息调度与AI处理.md` §2 MSG-REG-09 |

## 第三部分：实现完整性契约（Completeness Contract）

### CC-001：零占位原则

生成的代码中，以下内容一律视为未完成实现，不允许出现：
- `// TODO: 实现此方法`
- `throw new Error("Not implemented")`
- `return null;` 用于空数据（应返回 `undefined`）
- `// mock data` / `// fake implementation`

**唯一例外**：测试代码中的 Mock 数据（`src/test/` 目录下）。

### CC-002：伪代码完整翻译原则

详细设计文档第 4 节的每一个伪代码步骤，必须在代码中有对应实现。
如果某个步骤因信息不足无法实现，必须**停下来询问用户**。

### CC-003：外部依赖处理原则

| 情况 | 处理方式 |
|------|----------|
| 接口已在详设第 9 节定义 | 按文档定义实现调用代码（import 函数直接调用） |
| 接口未定义但调用逻辑清晰 | 生成接口定义 + 实现调用代码，标注「新增接口定义，请确认」 |
| 接口未定义且调用逻辑不清晰 | **停止生成，询问用户** |

### CC-004：业务规则完整性原则

详设第 2 节的每条业务规则，必须在代码中有对应实现。
生成代码后，必须逐条对照第 2 节规则，确认每条规则都有对应的代码实现。
